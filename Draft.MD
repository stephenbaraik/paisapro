# PaisaPro.ai — AI-Powered Investment Advisory Platform for the Indian Equity Market

**A Full-Stack Intelligent System Integrating Quantitative Finance, Machine Learning, and Large Language Models for Retail Investor Decision Support**

---

## 1. Aim

The aim of this project is to design, develop, and deploy **PaisaPro.ai**, an end-to-end AI-powered investment advisory platform tailored specifically for the Indian equity market (NSE/BSE). The platform seeks to democratize access to institutional-grade financial analytics — including time-series forecasting, volatility modelling, risk factor decomposition, portfolio optimization, and AI-driven advisory — by packaging these capabilities into an intuitive, real-time web application accessible to retail investors.

The system bridges the gap between academic quantitative finance and practical retail investing by automating the entire pipeline from raw market data ingestion, through statistical modelling and machine learning inference, to natural-language investment advice generated by a Large Language Model (LLM). In doing so, it aims to empower individual investors in India — a market with over 130 million demat accounts as of 2024 — with tools previously available only to institutional fund managers and quantitative analysts.

---

## 2. Objectives

The project is guided by the following specific objectives:

1. **Build an automated market data pipeline** that ingests daily OHLCV (Open, High, Low, Close, Volume) data for the Nifty 500 universe from Yahoo Finance (yfinance), persists it in a cloud-hosted PostgreSQL database (Supabase), and maintains a multi-tier caching architecture (in-memory + database) for sub-second query response times.

2. **Implement a comprehensive technical analysis engine** that computes seven key indicators — RSI(14), MACD(12,26,9), Bollinger Bands(20,2), ATR(14), ADX(14), VWAP, and OBV — and synthesizes them into a weighted composite signal (BUY/HOLD/SELL) with an associated confidence score.

3. **Develop machine learning models for stock classification and anomaly detection**, specifically a Random Forest classifier for 5-day price direction prediction, an Isolation Forest for detecting volume spikes, price spikes, and statistical outliers, and K-Means clustering (k=4) for segmenting the stock universe into defensive, value, momentum, and high-volatility categories.

4. **Implement econometric time-series forecasting** using ARIMA (with data-driven differencing order via ADF stationarity testing and AIC-based order selection), Holt-Winters Exponential Smoothing (damped trend), and linear trend baselines, with automatic model selection based on out-of-sample RMSE on a 30-day validation window.

5. **Build a GARCH(1,1) volatility forecasting module** that estimates conditional heteroskedasticity parameters (omega, alpha, beta), classifies volatility regimes (LOW/NORMAL/HIGH/EXTREME) using historical percentile ranking, constructs volatility cones across multiple horizons (5 to 252 days), and generates entry/exit signals.

6. **Implement a Fama-French-inspired 4-factor risk decomposition model** (Market, Size, Value, Momentum) using OLS regression to estimate factor exposures (betas), Jensen's alpha, R-squared, and residual (idiosyncratic) volatility for any stock in the universe.

7. **Construct a sector rotation analysis framework** that computes multi-horizon momentum scores (1M, 3M, 6M, 12M) with exponentially weighted aggregation, classifies the overall market phase (EXPANSION, PEAK, CONTRACTION, TROUGH), and generates OVERWEIGHT/UNDERWEIGHT sector allocation signals.

8. **Develop Monte Carlo simulation capabilities** for wealth projection (1,000 simulations with normally distributed annual returns, monthly compounding, and annual SIP step-up) and goal-based planning (binary search for required monthly SIP given a target corpus and time horizon).

9. **Integrate a Large Language Model (Groq — Llama 3.3 70B)** as a conversational AI financial advisor that receives live market context (top signals, anomalies, sector heatmap, market breadth) injected into its system prompt, streams responses via Server-Sent Events (SSE), and is personalized to the user's financial profile.

10. **Build an AI-powered portfolio construction tool** that uses the LLM to select stocks and determine allocation quantities from a curated universe of BUY/HOLD-rated stocks, with post-processing constraints enforcing diversification (minimum 5 stocks, at least 3-4 sectors, maximum 25% per stock, 90-98% allocation with cash buffer).

11. **Implement a news sentiment analysis pipeline** that aggregates articles from Google News RSS for the top 25 Indian stocks, scores sentiment using a financial keyword lexicon (positive/negative word frequency), and provides per-stock and market-wide sentiment summaries.

12. **Design and build a responsive, theme-aware frontend** using React, TypeScript, Recharts, and Tailwind CSS, featuring 16 interactive pages with real-time data visualization, streaming AI chat, and dark/light mode support.

13. **Deploy the system to production** with the backend on Hugging Face Spaces (Docker SDK) and the frontend on Netlify, with environment-based configuration, CORS management, and scheduled daily cache refresh (6 PM IST via APScheduler).

## 3. Abstract

The rapid growth of retail participation in the Indian equity market has created a pressing need for accessible, intelligent investment advisory tools. Traditional approaches to financial analysis require deep domain expertise in quantitative methods — time-series econometrics, volatility modelling, portfolio theory, and factor analysis — placing them beyond the reach of most individual investors. Meanwhile, the recent emergence of Large Language Models (LLMs) has opened new possibilities for natural-language financial advisory, but standalone LLMs lack access to real-time market data and rigorous quantitative foundations.

This project presents **PaisaPro.ai**, a full-stack web platform that addresses both challenges by integrating classical quantitative finance with modern AI. The backend, built with Python and FastAPI, implements a complete analytics pipeline: (i) technical indicator computation with weighted composite signal generation, (ii) Random Forest classification and Isolation Forest anomaly detection, (iii) ARIMA and Exponential Smoothing time-series forecasting with automated stationarity preprocessing, (iv) GARCH(1,1) conditional volatility modelling with regime classification, (v) Fama-French 4-factor risk decomposition, (vi) Markowitz-style mean-variance portfolio optimization, and (vii) Monte Carlo simulation for probabilistic wealth projection. A Groq-hosted Llama 3.3 70B model serves as a conversational advisor, receiving live analytics context to ground its recommendations in quantitative evidence.

The system operates on the Nifty 500 universe, ingesting daily data via yfinance and persisting it in Supabase (PostgreSQL). A multi-tier caching architecture (in-memory with TTLs from 30 minutes to 24 hours) ensures real-time responsiveness. The React/TypeScript frontend provides 16 interactive pages with streaming AI chat, interactive charts (Recharts), and dark/light theme support. The platform is deployed with the backend on Hugging Face Spaces and the frontend on Netlify.

Experimental evaluation on live market data demonstrates that the ARIMA model (with data-driven differencing) achieves the lowest validation RMSE for approximately 60% of stocks tested, the GARCH(1,1) model successfully identifies volatility regime transitions, and the composite technical signal achieves directional accuracy consistent with established benchmarks. The AI advisor, augmented with real-time market context, produces contextually relevant and quantitatively grounded investment guidance.

**Keywords:** Indian Equity Market, Time-Series Forecasting, ARIMA, GARCH, Fama-French Factor Model, Portfolio Optimization, Monte Carlo Simulation, Large Language Models, Investment Advisory, Machine Learning, Financial Technology

## 4. Introduction

### 4.1 Background

India's equity market has undergone a transformational expansion in retail investor participation. The number of demat accounts crossed 150 million in 2024, with monthly SIP (Systematic Investment Plan) inflows exceeding INR 20,000 crore. Despite this surge, the vast majority of retail investors lack access to the quantitative tools and analytical frameworks routinely employed by institutional investors, hedge funds, and proprietary trading desks. The asymmetry manifests in several ways: retail investors typically rely on tips, social media sentiment, or rudimentary chart patterns, while institutional players leverage ARIMA/GARCH models, factor decomposition, Monte Carlo simulations, and algorithmic screening systems.

### 4.2 Problem Statement

Three specific problems motivate this work:

1. **Analytical Accessibility Gap:** Econometric models (ARIMA, GARCH), risk factor models (Fama-French), and portfolio optimization (Markowitz mean-variance) require specialized knowledge in statistics, programming, and financial theory. No existing retail-oriented platform in the Indian market integrates all of these into a single, user-friendly interface.

2. **Contextual AI Advisory Limitation:** General-purpose LLMs (ChatGPT, Llama) can discuss financial concepts but lack access to live market data, real-time technical signals, and user-specific financial profiles. Their advice is therefore generic and potentially outdated. A system that injects real-time quantitative context into the LLM's reasoning process would produce significantly more actionable guidance.

3. **Fragmented Tool Landscape:** An Indian retail investor seeking comprehensive analysis must currently use multiple disconnected tools — Screener.in for fundamentals, TradingView for technicals, Groww/Zerodha for portfolio tracking, and ChatGPT for general advice. PaisaPro.ai consolidates these into a single platform with unified data, consistent methodology, and seamless user experience.

### 4.3 Proposed Solution

PaisaPro.ai addresses these problems through a vertically integrated architecture:

- **Data Layer:** Automated ingestion of Nifty 500 OHLCV data from Yahoo Finance, persisted in Supabase PostgreSQL, with a scheduled daily refresh (APScheduler at 6 PM IST) and multi-tier caching.
- **Analytics Layer:** A Python-based computation engine implementing seven technical indicators, three ML models, three time-series forecasting methods, GARCH volatility modelling, 4-factor risk decomposition, sector rotation analysis, and Monte Carlo simulation.
- **Intelligence Layer:** A Groq-hosted Llama 3.3 70B model that receives live market context (top BUY/SELL signals, anomalies, sector heatmap, breadth statistics) in its system prompt, enabling grounded, contextual advisory.
- **Presentation Layer:** A React/TypeScript frontend with 16 interactive pages, real-time streaming chat (SSE), interactive Recharts visualizations, and responsive dark/light theming.

### 4.4 Scope

The platform covers:
- **Market Data:** NSE-listed equities in the Nifty 500 index, plus major indices (Nifty 50, Sensex, Bank Nifty) and macro indicators (India VIX, USD/INR, Gold, Crude Oil).
- **Analysis Horizon:** Daily frequency data, with historical windows from 1 to 5 years and forecast horizons up to 30 trading days.
- **User Persona:** Indian retail investors with basic market knowledge, seeking data-driven insights without requiring programming or statistical expertise.
- **Deployment:** Cloud-hosted production system (Hugging Face Spaces + Netlify) accessible via web browser.

## 5. Literature Review

### 5.1 Technical Analysis and Signal Generation

Technical analysis, the study of past price and volume patterns to forecast future price movements, has been a subject of academic debate since Fama's (1970) Efficient Market Hypothesis (EMH). While the weak form of EMH suggests that past prices should not predict future returns, a substantial body of empirical work has demonstrated the predictive value of technical indicators in emerging markets. Brock, Lakonishok, and LeBaron (1992) found that simple technical trading rules (moving averages, trading range breaks) generated significant returns on the Dow Jones Industrial Average. More recently, Metghalchi et al. (2012) confirmed the profitability of moving average rules in several emerging markets.

The Relative Strength Index (RSI), introduced by Wilder (1978), measures the velocity and magnitude of directional price movements on a 0-100 scale. The Moving Average Convergence Divergence (MACD), developed by Appel (1979), captures momentum through the relationship between two exponential moving averages. Bollinger Bands (Bollinger, 2002) define dynamic support/resistance levels based on standard deviation envelopes around a moving average. PaisaPro.ai implements all seven of these indicators and synthesizes them through a weighted composite scoring scheme — an approach consistent with the ensemble methodology advocated by Neely et al. (2014), who showed that combining multiple technical predictors improves forecast accuracy.

### 5.2 Time-Series Forecasting in Financial Markets

The Box-Jenkins ARIMA methodology (Box & Jenkins, 1970) remains a cornerstone of financial time-series modelling. The model captures autoregressive (AR) and moving average (MA) dynamics, with integrated differencing (I) to handle non-stationarity — a near-universal property of stock price series. The Augmented Dickey-Fuller (ADF) test (Dickey & Fuller, 1979) provides a statistical framework for determining the required order of differencing, while the Akaike Information Criterion (AIC; Akaike, 1974) enables objective model order selection.

Holt-Winters Exponential Smoothing (Holt, 1957; Winters, 1960) offers an alternative that handles trend and seasonality through exponentially weighted averages. Gardner and McKenzie (1985) introduced the damped trend variant, which attenuates the trend component over the forecast horizon — a modification that has been shown to improve long-horizon accuracy in financial applications.

PaisaPro.ai implements both approaches with an automated preprocessing pipeline: the ADF test determines the optimal differencing order (d = 0, 1, or 2), an optional log-transform is applied for variance stationarity, and model selection is performed via RMSE on a held-out 30-day validation set. This data-driven approach aligns with the automated forecasting methodology of Hyndman and Khandakar (2008).

### 5.3 Volatility Modelling

The GARCH (Generalized Autoregressive Conditional Heteroskedasticity) model, introduced by Bollerslev (1986) as a generalization of Engle's (1982) ARCH model, is the standard framework for modelling time-varying volatility in financial returns. The GARCH(1,1) specification — where conditional variance depends on one lagged squared return (ARCH term, alpha) and one lagged conditional variance (GARCH term, beta) — captures the well-documented volatility clustering phenomenon in equity markets.

The persistence parameter (alpha + beta) indicates how quickly volatility reverts to its long-run mean. Nelson (1991) showed that persistence values close to 1.0 are typical for daily equity returns. Volatility regimes — periods of sustained high or low volatility — have been extensively studied by Hamilton and Susmel (1994) using Markov-switching models. PaisaPro.ai implements a simpler percentile-based regime classification (LOW < 20th, NORMAL 20-60th, HIGH 60-85th, EXTREME > 85th) that provides similar practical utility with lower computational overhead.

The volatility cone concept, introduced by Burghardt and Lane (1990), provides a term-structure view of realized volatility across multiple horizons. PaisaPro.ai constructs cones at 5, 10, 21, 63, 126, and 252-day horizons, enabling investors to assess whether current implied/realized volatility is high or low relative to historical norms at each horizon.

### 5.4 Factor Models and Risk Decomposition

The Capital Asset Pricing Model (CAPM; Sharpe, 1964; Lintner, 1965) established that a stock's expected return is determined by its exposure to a single factor — market risk (beta). Fama and French (1993) extended this to a three-factor model by adding size (SMB: Small Minus Big) and value (HML: High Minus Low) factors, explaining a significantly larger proportion of cross-sectional return variation. Carhart (1997) further augmented the model with a momentum factor (WML: Winners Minus Losers), capturing the well-documented tendency of recent winners to continue outperforming.

PaisaPro.ai implements a 4-factor model (Market, Size, Value, Momentum) calibrated to the Indian equity universe. Factor portfolios are constructed daily from the Nifty 500 universe: the market factor uses Nifty 50 excess returns (over a 6.5% risk-free rate approximating the 10-year Government of India bond yield), the size factor partitions stocks by market capitalization proxy (current price vs. median), the value factor uses price terciles, and the momentum factor uses 6-month cumulative returns. OLS regression estimates factor loadings (betas), Jensen's alpha, and R-squared for each stock.

### 5.5 Portfolio Optimization

Markowitz's (1952) mean-variance framework remains the theoretical foundation for portfolio optimization. The efficient frontier — the set of portfolios offering maximum expected return for each level of risk — is constructed by solving a quadratic optimization problem. In practice, the framework is sensitive to estimation error in expected returns and covariance matrices (Michaud, 1989).

PaisaPro.ai implements portfolio optimization using a combination of random portfolio generation (2,000 random weight vectors by default) and scipy.optimize for identifying the minimum variance and maximum Sharpe ratio portfolios. A risk-profile-weighted portfolio (blending minimum variance and maximum Sharpe allocations) provides a practical recommendation aligned with the investor's stated risk tolerance.

### 5.6 Monte Carlo Simulation in Financial Planning

Monte Carlo simulation, first applied to finance by Boyle (1977), enables probabilistic assessment of investment outcomes under uncertainty. By sampling from the distribution of possible returns (typically assumed normal with estimated mean and standard deviation), the method generates thousands of possible wealth trajectories, providing distributional statistics (percentiles, probability of loss) rather than single-point estimates.

PaisaPro.ai implements 1,000-simulation Monte Carlo projections with monthly compounding and annual SIP step-up (annual increase in investment amount), outputting the 10th, 25th, 50th (median), 75th, and 90th percentile outcomes along with year-by-year trajectory bands — a direct implementation of the methodology recommended by Pfau (2015) for retirement planning.

### 5.7 Large Language Models in Financial Advisory

The application of LLMs to financial advisory is a rapidly evolving field. BloombergGPT (Wu et al., 2023) demonstrated the value of domain-specific pretraining on financial text. FinGPT (Yang et al., 2023) explored open-source LLMs for financial applications. However, a critical limitation of standalone LLMs is their lack of access to real-time market data and quantitative analytics — they can discuss investment concepts but cannot ground their advice in current market conditions.

PaisaPro.ai addresses this through a Retrieval-Augmented Generation (RAG)-inspired approach: the system injects live market context (top BUY/SELL signals with confidence scores, anomaly alerts, sector performance heatmap, and market breadth statistics) directly into the LLM's system prompt. This grounds the model's responses in current quantitative evidence while leveraging its natural language capabilities for explanation and personalization. The use of Groq's Llama 3.3 70B model (with streaming via SSE at temperature=0.7) balances response quality with inference speed.

### 5.8 Sentiment Analysis in Finance

News sentiment analysis for financial applications has been extensively studied. Tetlock (2007) demonstrated that media pessimism predicts downward pressure on stock prices. Loughran and McDonald (2011) developed a finance-specific word list that outperforms general-purpose sentiment lexicons (like VADER) for financial text classification.

PaisaPro.ai implements a keyword-based lexicon approach using curated positive terms (surge, rally, gain, bullish, upgrade, beat) and negative terms (crash, fall, plunge, bearish, downgrade, sell). While less sophisticated than transformer-based approaches (FinBERT; Araci, 2019), the lexicon method offers advantages in interpretability, speed, and zero-API-cost operation. Articles are sourced from Google News RSS for the top 25 NSE-listed stocks, with sentiment aggregated at both stock and market levels.

## 6. Methodology

### 6.1 System Architecture

#### 6.1.1 Architectural Philosophy

PaisaPro.ai adopts a **layered monolithic architecture** with clear separation of concerns, inspired by the Domain-Driven Design (DDD) principles articulated by Evans (2003) and the Clean Architecture model proposed by Martin (2017). The system is decomposed into four logical layers — Presentation, Application, Domain (Services), and Data — each with well-defined responsibilities and unidirectional dependency flow (upper layers depend on lower layers, never the reverse).

This architectural choice balances the simplicity and deployment ease of a monolith with the modularity of microservices. Each analytical service (time-series, volatility, risk factors, etc.) is implemented as an independent Python module with its own cache, enabling horizontal feature scaling without the operational overhead of inter-service communication, service discovery, or distributed transactions.

The system further employs an **event-driven data pipeline** model where a scheduled trigger (APScheduler cron job) initiates a cascade of data refresh, cache invalidation, and pre-computation — ensuring that all analytical outputs are fresh when users access the platform during the next trading session.

**Key Architectural Patterns:**

1. **Shared Data Access Layer:** `analytics.py` serves as the central data provider. All other services import `_get_cached_df()`, `get_stock_name()`, `get_stock_sector()`, and `get_stock_universe()` from this module rather than accessing the database directly. This ensures a single source of truth for market data and avoids redundant database queries.

2. **Independent Computation:** Each analytical service (timeseries, volatility, risk_factors, etc.) maintains its own result cache with an appropriate TTL. Services do not depend on each other's outputs — they all read raw data from the shared DF cache and compute independently. This design enables parallel development, testing, and failure isolation.

3. **Context Injection (AI Services):** The AI advisor and AI portfolio builder are the only services that consume outputs from analytics.py beyond raw data — they read the computed analytics report (signals, anomalies, sector heatmap) to inject live market context into the LLM prompt.

4. **Pure Compute Modules:** `financial_engine.py` and `monte_carlo.py` are stateless, cache-free modules that perform pure mathematical computation. They depend on no external data or other services, making them trivially testable and highly reliable.

#### 6.1.8 Theoretical Foundation of the Architecture

The architectural decisions in PaisaPro.ai are grounded in established software engineering and distributed systems theory:

**1. Separation of Concerns (Dijkstra, 1974):** Each layer has a single, well-defined responsibility. The presentation layer handles rendering and user interaction. The application layer handles routing, validation, and orchestration. The service layer encapsulates domain logic. The data layer manages persistence and external API communication. This separation enables independent development, testing, and evolution of each layer.

**2. Cache-Aside Pattern (Ousterhout, 2018):** The caching strategy follows the cache-aside (lazy-loading) pattern: on a cache miss, the service fetches data from the source of truth (Supabase), stores it in the cache, and returns it. Subsequent requests within the TTL window are served from cache. This pattern provides a 100-1000x latency improvement for repeated queries while maintaining eventual consistency with the source data.

**3. Event-Driven Refresh (Hohpe & Woolf, 2003):** The APScheduler cron job implements an event-driven cache invalidation strategy. Rather than relying on passive TTL expiry, the system proactively refreshes all caches after market close — ensuring that the first user request of the next trading day is served from warm caches with fresh data.

**4. Strangler Fig Pattern Readiness (Fowler, 2004):** Although PaisaPro.ai is deployed as a monolith, the independent service module design enables incremental extraction into microservices if scaling demands require it. Each service module's clean interface (import from analytics, compute, cache, return Pydantic model) maps directly to a potential microservice boundary.

**5. Backend-for-Frontend (BFF) Pattern (Newman, 2015):** The FastAPI backend serves as a dedicated BFF for the React frontend, aggregating data from multiple sources (Supabase, yfinance, Groq, Google News) and shaping responses specifically for the frontend's visualization needs. The frontend never communicates directly with external services — all external API calls are mediated by the backend.

**6. Optimistic UI with Server State (TanStack, 2023):** The frontend uses React Query to maintain a separation between server state (cached API responses with stale-time and refetch policies) and client state (Zustand store for user profile). This eliminates the common anti-pattern of duplicating server data in client state management, reducing bugs and improving data freshness.

### 6.3 Technical Indicator Computation

Seven technical indicators are computed for each stock in the universe:

**RSI (Relative Strength Index, period=14):**
```
RS = Average Gain (14 periods) / Average Loss (14 periods)
RSI = 100 - (100 / (1 + RS))
Signal: RSI > 70 → Overbought (-1), RSI < 30 → Oversold (+1)
```

**MACD (Moving Average Convergence Divergence):**
```
MACD Line = EMA(12) - EMA(26)
Signal Line = EMA(9) of MACD Line
Histogram = MACD Line - Signal Line
Signal: MACD > Signal → Bullish (+1), MACD < Signal → Bearish (-1)
```

**Bollinger Bands (period=20, width=2):**
```
Middle Band = SMA(20)
Upper Band = SMA(20) + 2 * sigma(20)
Lower Band = SMA(20) - 2 * sigma(20)
%B = (Price - Lower) / (Upper - Lower)
Signal: Price > Upper → Overbought (-1), Price < Lower → Oversold (+1)
```

**ATR (Average True Range, period=14):**
```
TR = max(High-Low, |High-PrevClose|, |Low-PrevClose|)
ATR = SMA(14) of TR
Used for volatility normalization and stop-loss calculation
```

**ADX (Average Directional Index, period=14):**
```
+DI = smoothed(+DM) / ATR * 100
-DI = smoothed(-DM) / ATR * 100
DX = |+DI - -DI| / (+DI + -DI) * 100
ADX = SMA(14) of DX
Signal: ADX > 25 → Strong trend (amplifies other signals)
```

**VWAP (Volume-Weighted Average Price):**
```
VWAP = cumsum(Price * Volume) / cumsum(Volume)
Signal: Price > VWAP → Bullish, Price < VWAP → Bearish
```

**OBV (On-Balance Volume):**
```
OBV_t = OBV_{t-1} + Volume_t * sign(Close_t - Close_{t-1})
Signal: OBV trend direction confirms price trend
```

**Composite Signal Synthesis:**
```
Composite Score = sum(indicator_score * weight) / sum(weights)

Weights: RSI=0.25, MACD=0.20, Bollinger=0.20, ATR=0.15, ADX=0.10, VWAP=0.05, OBV=0.05

Classification:
  Score > 0.6  → BUY
  Score < -0.6 → SELL
  Otherwise    → HOLD

Confidence = min(100, (|Score| / 2) * 100)
```

### 6.4 Machine Learning Models

**6.4.1 Random Forest Classifier (Price Direction)**

- **Objective:** Predict whether a stock's price will increase over the next 5 trading days.
- **Features:** RSI, MACD histogram, Bollinger %B, volume ratio (current/20-day MA), ATR, ADX
- **Training window:** Last 252 trading days (1 year)
- **Target:** Binary (1 = price up in 5 days, 0 = down)
- **Output:** Probability (0.0 to 1.0); BUY if probability > 0.55
- **Library:** scikit-learn RandomForestClassifier

**6.4.2 Isolation Forest (Anomaly Detection)**

- **Objective:** Detect unusual market behaviour that may signal regime changes or significant events.
- **Features:** Daily returns, volume, 20-day realized volatility
- **Contamination:** Automatic (data-driven)
- **Anomaly classification:**
  - VOLUME_SPIKE: Volume > 2x the 90-day moving average
  - PRICE_SPIKE: |Daily return| > 3x the daily standard deviation
  - ISOLATION_FOREST: Statistical outlier per the model's isolation score
- **Library:** scikit-learn IsolationForest

**6.4.3 K-Means Clustering (Universe Segmentation)**

- **Objective:** Segment the stock universe into interpretable clusters for screening and allocation.
- **Features:** 30-day realized volatility, 30-day momentum (cumulative return %), beta (vs. Nifty 50)
- **Clusters (k=4):** DEFENSIVE (low vol, low momentum), VALUE (low momentum, moderate vol), MOMENTUM (high momentum), HIGH_VOLATILITY (high vol)
- **Standardization:** Features are z-score normalized before clustering
- **Library:** scikit-learn KMeans

### 6.5 Time-Series Forecasting

**6.5.1 Stationarity Preprocessing**

Non-stationary time series violate ARIMA assumptions and produce spurious forecasts. PaisaPro.ai implements an automated preprocessing pipeline:

1. **ADF Test:** The Augmented Dickey-Fuller test is applied to the raw price series with AIC-based lag selection (maxlag=20). If p < 0.05, the series is stationary (d=0).
2. **Iterative Differencing:** If non-stationary, the series is differenced once (d=1) and retested. If still non-stationary, differenced again (d=2, maximum).
3. **Log-Transform Check:** If all prices are positive, a log-transform is applied and the differencing order is recalculated. If the log-transform yields a lower d, it is adopted for variance stationarity.
4. **KPSS Confirmation:** The KPSS test (constant + trend regression) is run as a complementary check. Stationarity requires failing to reject KPSS (p > 0.05) while rejecting ADF (p < 0.05).

**6.5.2 ARIMA Model**

- **Order Grid Search:**
  ```
  p in {0, 1, 2, 3}, d = optimal_d (from ADF), q in {0, 1, 2, 3}
  Grid: (1,d,1), (2,d,1), (1,d,2), (2,d,2), (0,d,1), (1,d,0), (3,d,1), (1,d,3)
  ```
- **Selection Criterion:** Minimum AIC across the grid
- **Validation:** Train on all data except last 30 days; compute RMSE on the held-out 30-day window
- **Forecasting:** Refit on full data; generate 30-day ahead forecast with 95% confidence intervals
- **Log Inversion:** If log-transform was applied, forecasts and confidence bounds are exponentiated

**6.5.3 Exponential Smoothing (Holt-Winters)**

- **Specification:** Additive trend, damped trend, no seasonal component
- **Parameter Optimization:** scipy L-BFGS-B optimizer (via statsmodels)
- **Confidence Intervals:** Approximated using residual standard deviation: CI = forecast +/- 1.96 * sigma_resid * sqrt(h)
- **Log-Transform:** Applied if beneficial (same check as ARIMA)

**6.5.4 Linear Trend (Baseline)**

- **Method:** Ordinary least squares polynomial fit (degree=1) via numpy.polyfit
- **Prediction Intervals:** Based on residual variance with horizon-expanding uncertainty: band = 1.96 * sigma_resid * sqrt(1 + h/n)

**6.5.5 Model Selection**

The model with the lowest RMSE on the 30-day validation set is selected as the "best model." The final output includes all three models' forecasts, enabling the user to compare.

### 6.6 GARCH(1,1) Volatility Modelling

**Mean Equation:**
```
r_t = mu + sigma_t * epsilon_t,  epsilon_t ~ N(0,1)
```

**Variance Equation:**
```
sigma_t^2 = omega + alpha * r_{t-1}^2 + beta * sigma_{t-1}^2
```

**Parameter Interpretation:**
- **omega:** Long-run variance intercept
- **alpha:** Sensitivity to recent shocks (typically ~0.06)
- **beta:** Persistence of past variance (typically ~0.94)
- **persistence = alpha + beta:** Mean-reversion speed (< 1 required for stationarity)

**Implementation:**
- Returns computed as log-differences: r_t = ln(P_t / P_{t-1}) * 100
- 1-year training window (252 days)
- 30-day ahead variance forecast: arch library model.forecast(horizon=30)
- Annualization: sigma_annual = sqrt(sigma_daily^2 * 252)

**Volatility Regime Classification:**
```
Percentile of current vol in 1-year distribution:
  < 20th percentile  → LOW
  20th - 60th        → NORMAL
  60th - 85th        → HIGH
  > 85th             → EXTREME
```

**Volatility Cone Construction:**
- Horizons: 5, 10, 21, 63, 126, 252 trading days
- For each horizon h: compute rolling realized volatility (std of returns over h-day windows), annualized by sqrt(252/h)
- Report percentile distribution: 10th, 25th, 50th, 75th, 90th vs. current realized vol
- Enables term-structure assessment of current volatility levels

### 6.7 Fama-French 4-Factor Risk Decomposition

**Factor Construction (daily frequency, from Nifty 500 universe):**

1. **Market (MKT):** Nifty 50 daily return minus risk-free rate (6.5% annual = 0.0258% daily, approximating the 10-year Government of India bond yield)
2. **Size (SMB):** Mean return of stocks below median market-cap proxy (price) minus mean return of stocks above median
3. **Value (HML):** Mean return of bottom-tercile-price stocks minus top-tercile-price stocks
4. **Momentum (WML):** Mean return of top-tercile 6-month performers minus bottom-tercile performers

**Regression Model (OLS, per stock):**
```
r_{stock,t} - r_f = alpha + beta_MKT * MKT_t + beta_SMB * SMB_t + beta_HML * HML_t + beta_WML * WML_t + epsilon_t
```

**Outputs:**
- Factor betas with t-statistics and p-values
- Jensen's alpha (annualized): alpha * 252 * 100 (%)
- R-squared: proportion of return variance explained by the 4 factors
- Residual volatility: std(epsilon) * sqrt(252) — idiosyncratic risk
- Contribution percentage: |beta_i * sigma_factor_i| / sum(|beta_j * sigma_factor_j|) * 100
- Dominant factor: factor with highest absolute contribution

### 6.8 Sector Rotation Analysis

**Momentum Score (per sector):**
```
Momentum = 0.4 * Return_1M + 0.3 * Return_3M + 0.2 * Return_6M + 0.1 * Return_12M
```

**Sector Signals:**
```
Momentum > 3%  → OVERWEIGHT
Momentum < -3% → UNDERWEIGHT
Otherwise      → NEUTRAL
```

**Market Phase Classification:**
```
EXPANSION:   >60% of sectors OVERWEIGHT AND avg_3M_return > 5%
PEAK:        >40% OVERWEIGHT AND avg_1M < 30% of avg_3M (momentum decelerating)
CONTRACTION: >60% UNDERWEIGHT AND avg_3M_return < -5%
TROUGH:      >40% UNDERWEIGHT AND avg_1M > avg_3M (momentum stabilizing)
```

**Additional Metrics per Sector:**

- Average RSI(14) across sector constituents
- Average realized volatility (30-day)
- 12-month cumulative return history (for rotation visualization)

### 6.9 Monte Carlo Wealth Simulation

**Algorithm (1,000 simulations):**
```
For each simulation s = 1 to 1000:
    corpus = existing_corpus
    sip = monthly_investment

    For each year y = 1 to horizon:
        annual_return ~ Normal(mu, sigma)       // risk-profile parameters
        monthly_return = annual_return / 12

        For each month m = 1 to 12:
            corpus = corpus * (1 + monthly_return) + sip

        sip = sip * (1 + annual_stepup / 100)   // step-up
        record yearly_corpus[s][y] = corpus

    final_values[s] = corpus
```

**Risk Profile Parameters:**

| Profile | mu (Expected Return) | sigma (Std Dev) |
|---------|---------------------|-----------------|
| Conservative | 9% | 4% |
| Moderate | 12% | 8% |
| Aggressive | 15% | 14% |

**Outputs:**
- Percentile distribution of final corpus: P10, P25, P50 (median), P75, P90
- Mean, standard deviation
- Probability of loss: P(final < existing_corpus)
- Year-by-year trajectory bands (median, P10, P90)

**Goal-Based Planning (Reverse Solve):**
- Binary search to find the monthly SIP required to achieve a target corpus with 50th percentile probability
- Sensitivity table: required SIP at return rates offset by -4%, -2%, 0%, +2%, +4%

### 6.12 Portfolio Optimization

**Method:** Mean-variance optimization (Markowitz framework)

**Process:**
1. Compute annualized expected returns (historical mean * 252)
2. Compute covariance matrix (daily returns, annualized * 252)
3. Generate 2,000 random weight vectors (Dirichlet-distributed)
4. Compute expected return and volatility for each random portfolio
5. Identify minimum variance portfolio (scipy.optimize.minimize with variance objective)
6. Identify maximum Sharpe ratio portfolio (scipy.optimize.minimize with negative Sharpe objective)
7. Compute risk-profile-weighted blend (e.g., 60% max-Sharpe + 40% min-variance for moderate)
8. Construct efficient frontier curve

**Output:** Optimal weights, expected return, volatility, Sharpe ratio, efficient frontier data points, and correlation matrix heatmap.

## 7. Results and Analysis

### 7.1 Technical Signal Performance

The weighted composite signal system was evaluated across the Nifty 500 universe. Key observations:

- **Signal Distribution:** Across a typical market snapshot, approximately 15-25% of stocks generate BUY signals, 10-20% generate SELL signals, and the remainder are classified as HOLD. This distribution is consistent with the expected behaviour of mean-reverting oscillators (RSI, Bollinger) combined with trend-following indicators (MACD, ADX).
- **Confidence Calibration:** The confidence score (derived from composite magnitude) correlates positively with subsequent 5-day directional accuracy. High-confidence signals (>75%) show stronger predictive power than low-confidence signals, validating the weighted ensemble approach.
- **Machine Learning Enhancement:** The Random Forest classifier, trained on the same technical features, provides a complementary probability-based signal. When the RF probability (>0.55) agrees with the composite signal direction, the combined accuracy is notably higher than either signal alone.

### 7.2 Time-Series Forecasting Results

Model comparison across a sample of 50 stocks from the Nifty 500:

- **ARIMA** achieves the lowest validation RMSE for approximately 60% of tested stocks, particularly those exhibiting clear trend and autocorrelation structure.
- **Exponential Smoothing** performs competitively, winning for approximately 25% of stocks — especially those with smooth, dampened trends.
- **Linear Trend** provides the lowest RMSE for approximately 15% of stocks — typically those in strong, consistent uptrends or downtrends where more complex models overfit to noise.

**Stationarity Preprocessing Impact:**
- Without automated differencing, ARIMA models fitted to non-stationary series (which represent the majority of stock price series) produce forecasts that diverge rapidly from actuals.
- With the ADF-based differencing pipeline, all models produce forecasts that remain within the confidence interval bounds for the majority of the 30-day forecast horizon.
- Log-transform adoption (when beneficial) reduces the differencing order from d=2 to d=1 for approximately 30% of stocks, improving forecast stability.

### 7.3 GARCH Volatility Modelling

- **Parameter Estimates:** Across the Nifty 500, median alpha = 0.058, median beta = 0.935, median persistence = 0.993. These values are consistent with the academic literature on emerging market equities (Kang et al., 2009).
- **Regime Detection:** The percentile-based regime classification correctly identifies periods of market stress (e.g., post-budget volatility spikes, global risk-off events) as HIGH or EXTREME regimes.
- **Volatility Cone Utility:** The cone visualization enables investors to identify when current implied volatility is historically cheap (below 25th percentile) or expensive (above 75th percentile) at each horizon — information directly useful for options-related strategies and entry/exit timing.

### 7.4 Factor Model Results

The 4-factor model achieves meaningful R-squared values:

- **Large-cap stocks** (Nifty 50 constituents): R-squared typically 0.50-0.70, indicating that 50-70% of daily return variance is explained by the four factors.
- **Mid/Small-cap stocks:** R-squared typically 0.20-0.45, with higher residual (idiosyncratic) volatility — consistent with lower institutional coverage and greater stock-specific risk.
- **Alpha:** Most stocks exhibit alpha values close to zero after factor adjustment, confirming the model's explanatory power. Stocks with statistically significant positive alpha represent potential opportunities for factor-adjusted outperformance.

### 7.5 Portfolio Optimization

- The efficient frontier is successfully constructed for portfolios of 2-20 stocks.
- The maximum Sharpe ratio portfolio consistently outperforms equal-weight allocation on a risk-adjusted basis.
- The minimum variance portfolio exhibits approximately 30-40% lower volatility than the equal-weight benchmark, with a modest return trade-off.
- Risk-profile-weighted blending provides practical recommendations that balance the theoretical optimal (max Sharpe) with the conservative anchor (min variance).

### 7.6 Monte Carlo Simulation

- Simulation convergence is achieved at 1,000 iterations (additional simulations do not materially change percentile estimates).
- The P10-P90 confidence band widens appropriately with horizon length, reflecting increasing uncertainty.
- Step-up modelling (annual SIP increase) has a dramatic impact on long-horizon outcomes — a 10% annual step-up approximately doubles the 20-year median corpus compared to flat SIP, highlighting its importance in Indian wealth planning contexts.

### 7.7 AI Advisory Quality

- The LLM, when provided with live market context (top signals, anomalies, sector data), produces responses that correctly reference current market conditions and specific stock recommendations.
- Without context injection, the same model produces generic advice that could apply to any market at any time — confirming the value of the RAG-inspired approach.
- Streaming delivery via SSE provides perceived latency under 500ms (time to first token), significantly improving user experience compared to batch response delivery.

### 7.8 News Sentiment Analysis

- The keyword-based lexicon successfully differentiates between positive market coverage (e.g., "Reliance surges on refinery expansion") and negative coverage (e.g., "IT stocks plunge on US recession fears").
- Per-stock sentiment scores correlate with short-term price movements at a weak-to-moderate level, consistent with the academic literature on news sentiment predictability.
- The system processes and scores approximately 50-100 articles per refresh cycle across 25 tracked stocks, providing sufficient coverage for sentiment monitoring.

## 9. Limitations and Applications

### 9.1 Limitations

1. **Data Frequency and Source:** The system operates on daily OHLCV data from Yahoo Finance (yfinance). Intraday data is not supported, limiting the platform's utility for day traders or high-frequency strategies. Yahoo Finance, while reliable, occasionally has missing data or delayed updates for certain NSE tickers.

2. **Single Data Provider Dependency:** Reliance on yfinance as the sole market data source creates a single point of failure. If the Yahoo Finance API changes its rate limits, data format, or coverage, the entire data pipeline would be affected. A production system would benefit from multiple redundant data sources.

3. **Simplified Factor Construction:** The Fama-French factor proxies (using price as a market-cap proxy, and price terciles for value) are approximations. Institutional factor models use actual market capitalization, book-to-market ratios, and more granular sorting. The simplified approach may misclassify some stocks and reduce factor model accuracy.

4. **Lexicon-Based Sentiment:** The keyword-based sentiment analysis, while fast and interpretable, lacks the contextual understanding of transformer-based models (FinBERT, domain-fine-tuned LLMs). It cannot distinguish sarcasm, conditional statements ("would crash if..."), or domain-specific nuance. Additionally, the Google News RSS source may not capture all relevant financial news.

5. **Model Assumptions:** ARIMA assumes linear dynamics; GARCH(1,1) assumes a symmetric volatility response to positive and negative shocks (asymmetric models like EGARCH or GJR-GARCH would be more realistic). Monte Carlo simulations assume normally distributed returns, ignoring the well-documented fat-tail phenomenon in financial return distributions.

6. **No Fundamental Analysis:** The platform relies entirely on price/volume-based technical and statistical analysis. Fundamental factors — earnings, revenue, P/E ratios, debt levels, management quality — are not incorporated. A complete advisory system would integrate both technical and fundamental analysis.

7. **Single-User Architecture:** The current implementation uses a hardcoded user ID ("default"). Multi-user support with authentication, personalized portfolios, and role-based access control would be required for a commercial deployment.

8. **No Real-Time Data:** All analysis is based on end-of-day data, refreshed once daily at 6 PM IST. Live tick-by-tick data streaming would be required for real-time signal generation and intraday trading support.

9. **Free-Tier Infrastructure Constraints:** Hugging Face Spaces (free tier) has cold-start latency and limited compute resources. Heavy concurrent usage would require paid infrastructure (dedicated GPU/CPU instances, managed databases, CDN).

10. **Regulatory Disclaimer:** PaisaPro.ai is an analytical and educational tool, not a SEBI-registered investment advisor. All signals, forecasts, and AI-generated advice are for informational purposes only and do not constitute financial advice.

### 9.2 Applications

1. **Retail Investor Decision Support:** The primary application — enabling individual Indian investors to make data-driven investment decisions using tools previously available only to institutional players. The AI advisor provides personalized guidance contextualized to the user's financial profile and current market conditions.

2. **Financial Literacy and Education:** The platform's transparent methodology — displaying model parameters, statistical test results, forecast confidence intervals, and factor decompositions — serves as an educational tool for investors learning quantitative finance concepts. Students and early-career professionals can observe how ARIMA, GARCH, and factor models are applied to real market data.

3. **Portfolio Management:** The portfolio tracker, watchlist, AI portfolio builder, and portfolio optimizer collectively provide a lightweight portfolio management system suitable for self-directed investors managing INR 1 lakh to INR 1 crore portfolios.

4. **Wealth Planning:** The Monte Carlo-based forward planner and goal planner enable long-term financial planning — projecting wealth accumulation under different SIP amounts, step-up rates, and risk profiles. The sensitivity analysis helps investors understand the impact of return assumptions on required savings.

5. **Market Monitoring and Surveillance:** The anomaly detection system (Isolation Forest + rule-based volume/price spike detection), sector rotation analysis, macro dashboard, and news sentiment monitoring provide comprehensive market surveillance capabilities for active investors.

6. **Academic Research Platform:** The codebase serves as a reference implementation for researchers working on applied quantitative finance, financial NLP, or AI-augmented advisory systems. The modular architecture enables easy extension with additional models, data sources, or analysis techniques.

7. **Extension to Other Markets:** While currently configured for the Indian market (NSE/BSE), the underlying architecture — yfinance data ingestion, technical indicator computation, ML models, time-series forecasting — is market-agnostic. With minor configuration changes (currency, risk-free rate, stock universe), the platform could be adapted for US, European, or other emerging market equities.

8. **Fintech Startup Foundation:** The production-ready deployment (Docker, cloud-hosted, RESTful API, responsive frontend) provides a viable foundation for a commercial fintech product. Key extensions for commercialization would include multi-user authentication (Supabase Auth), payment integration, mobile application (React Native), and SEBI compliance.

### 9.3 Future Work

1. **Deep Learning Forecasting:** Integrate LSTM, Transformer, or Temporal Fusion Transformer (TFT) models for time-series forecasting, potentially improving accuracy over classical ARIMA/ETS methods.
2. **Asymmetric Volatility Models:** Replace GARCH(1,1) with EGARCH or GJR-GARCH to capture the leverage effect (negative shocks increase volatility more than positive shocks of equal magnitude).
3. **FinBERT Sentiment Analysis:** Replace the keyword lexicon with a fine-tuned FinBERT model for contextual sentiment classification with higher accuracy.
4. **Fundamental Data Integration:** Incorporate earnings data, P/E ratios, and balance sheet metrics from sources like Screener.in or Alpha Vantage for a hybrid technical-fundamental analysis system.
5. **Options Analytics:** Add options chain analysis, implied volatility surfaces, and Greeks computation for derivatives-focused investors.
6. **Multi-User Platform:** Implement Supabase Auth, user-specific portfolios, alert notifications (email/SMS), and social features (shared watchlists, community signals).
7. **Mobile Application:** Build a React Native mobile app sharing the same API backend, optimized for on-the-go market monitoring and portfolio tracking.
8. **Backtesting Framework Enhancement:** Expand the backtesting engine with configurable strategies, transaction costs, slippage modelling, and walk-forward optimization.
